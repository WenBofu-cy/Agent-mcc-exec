# ag-mcc-12-执行日志记录单元 接口规格

---

## 基本信息

| 项 | 内容 |
|----|------|
| 模块编号 | ag-mcc-12 |
| 模块名称 | 执行日志记录单元 |
| 所属分区 | 四、反馈与日志 |
| 核心职责 | 作为 MCC 行动执行层的全链路执行日志记录中枢，接收来自各执行模块（ag-mcc-01至ag-mcc-11）推送的执行事件、状态变更、异常告警等日志数据。采用追加写模式将所有日志条目持久化存储，禁止修改或删除已落盘的日志。存储周期不少于 3 年，为事故追溯、性能分析、合规审计提供完整的数据基础。支持按时间、指令ID、工具类型、执行状态等多维度检索。不参与任何工具调用或决策，仅负责日志的记录、存储与检索服务 |
| 依赖模块 | ag-mcc-01（执行调度核心，推送指令接收与分发日志）、ag-mcc-02（工具超时管理器，推送超时事件日志）、ag-mcc-03（资源配额管控单元，推送资源限流日志）、ag-mcc-06/07/08（各执行引擎，推送执行结果日志）、ag-mcc-09（结果校验器，推送校验结果日志）、ag-mcc-10（执行偏差监控，推送偏差事件日志）、ag-mcc-11（闭环反馈单元，推送闭环事件日志） |
| 被依赖模块 | ag-ecc-12（资源调度模块，可通过 CerebellumBus 查询执行日志供审计）、ag-mcc-01（查询历史执行日志）、系统管理接口（日志导出与离线分析） |


## 内部状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|----------|
| 正常记录 | `NORMAL_LOGGING` | 日志存储正常，接收并记录各模块推送的日志事件 | 系统初始化完成，存储介质就绪 |
| 存储空间不足 | `LOW_STORAGE` | 日志存储空间低于警戒阈值 | 剩余存储空间 < 15% |
| 存储故障 | `STORAGE_FAULT` | 存储介质写入失败或不可用 | 连续写入失败 ≥ 3 次 |
| 暂停服务 | `SYSTEM_PAUSED` | 系统紧急熔断 | 收到紧急熔断指令 |

## 日志条目结构与事件类型

### 一、统一日志条目结构

| 字段名称 | 数据类型 | 说明 |
|----------|----------|------|
| 日志条目ID | String | 唯一标识，格式：MCC-LOG-{序列号}-{时间戳} |
| 序列号 | UInt64 | 全局单调递增序列号，确保时序可追溯 |
| 事件类型 | Enum | 指令接收/指令分发/执行结果/超时事件/资源限流/校验结果/偏差事件/闭环事件 |
| 指令ID | String | 关联的工具调用指令唯一标识 |
| 来源模块编号 | String | 推送该日志的模块编号 |
| 工具名称 | String | 被调用的工具名称 |
| 工具类型 | Enum | API/CODE/FILE/LLM/DB |
| 执行状态 | Enum | 成功/失败/超时/异常（仅执行结果类日志） |
| 详细数据 | Struct | 事件类型特定的详细数据负载 |
| UTC 时间戳 | Float | 事件发生的精确 UTC 时间戳 |
| 校验签名 | String | 日志条目的 HMAC 签名，用于防篡改验证 |

### 二、事件类型特定的详细数据

| 事件类型 | 详细数据包含 |
|----------|------|
| 指令接收 | 优先级、接收时间戳 |
| 指令分发 | 目标执行模块、分发时间戳、队列等待时长 |
| 执行结果 | 实际耗时、输出数据大小、错误码、资源消耗（API次数/内存MB/Token数） |
| 超时事件 | 超时阈值、实际耗时 |
| 资源限流 | 限流类型、当前使用量、配额上限 |
| 校验结果 | 校验标记、校验详情、校验耗时 |
| 偏差事件 | 偏差类型、综合偏差度、告警等级 |
| 闭环事件 | 闭环状态、各环节耗时、异常标记 |


## 核心处理逻辑

```
FUNCTION execution_logger_main_loop():
    STATE_NORMAL = NORMAL_LOGGING
    STATE_LOW = LOW_STORAGE
    STATE_FAULT = STORAGE_FAULT
    STATE_PAUSED = SYSTEM_PAUSED

    SET internal_state = STATE_NORMAL
    初始化日志存储（追加写文件或数据库）
    初始化全局序列号 = 0
    初始化内存缓冲队列（用于存储故障时的暂存）
    写入失败计数器 = 0

    WHILE 系统运行中:
        // 第1步：紧急熔断
        IF 收到紧急熔断指令:
            SET internal_state = STATE_PAUSED
            尝试强制刷写内存缓冲队列到存储
            CONTINUE
        ELSE IF 收到恢复指令 AND internal_state == STATE_PAUSED:
            SET internal_state = STATE_NORMAL

        // 第2步：接收并记录各类日志事件
        FOR EACH 日志事件 IN 获取所有待记录事件():
            全局序列号++
            当前时间戳 = NOW()

            // 2a. 构建统一日志条目
            日志条目 = 构建日志条目(
                日志条目ID="MCC-LOG-" + 全局序列号 + "-" + 当前时间戳,
                序列号=全局序列号,
                事件类型=日志事件.事件类型,
                指令ID=日志事件.指令ID,
                来源模块编号=日志事件.来源模块编号,
                工具名称=日志事件.工具名称,
                工具类型=日志事件.工具类型,
                执行状态=日志事件.执行状态,
                详细数据=日志事件.详细数据,
                UTC时间戳=当前时间戳,
                校验签名=HMAC签名(上述字段)
            )

            // 2b. 持久化写入
            IF internal_state == STATE_FAULT:
                暂存至内存缓冲队列()
                IF 内存缓冲队列.size > 5000:
                    丢弃最旧的日志条目（记录告警）
            ELSE:
                写入结果 = 日志存储.追加写入(日志条目)
                IF 写入结果.失败:
                    写入失败计数器++
                    IF 写入失败计数器 >= 3:
                        SET internal_state = STATE_FAULT
                        向 ag-mcc-01 发送存储告警("存储故障", "连续写入失败3次")
                        暂存至内存缓冲队列()
                ELSE:
                    写入失败计数器 = 0
                    回写内存缓冲队列()

        // 第3步：存储空间管理
        IF internal_state != STATE_FAULT:
            存储状态 = 查询日志存储状态()
            剩余比例 = 存储状态.剩余容量 / 存储状态.总容量

            IF 剩余比例 < 10%:
                SET internal_state = STATE_LOW
                触发旧日志清理(保留最近 3 年数据)
                向 ag-mcc-01 发送存储告警("空间不足", "日志存储空间低于10%")
            ELSE IF 剩余比例 < 5%:
                触发强制清理(保留最近 2 年数据)
                向 ag-mcc-01 发送存储告警("空间严重不足", "日志存储空间低于5%，已触发强制清理")

        // 第4步：响应日志查询请求
        IF 收到日志查询请求:
            匹配日志列表 = 日志存储.按条件检索(请求.查询条件)
            向请求来源返回日志查询结果(匹配日志列表)

        // 第5步：响应日志导出请求
        IF 收到日志导出请求:
            导出日志列表 = 日志存储.按条件检索(请求.导出范围)
            日志导出包 = 构建日志导出包(导出日志列表)
            向请求来源返回日志导出包

        // 第6步：周期性状态上报
        IF 距上次状态上报 >= 60秒:
            向 ag-mcc-01 上报存储状态

        SLEEP 10ms
```


## 约束与异常处理

| 场景 | 处理方式 | 恢复条件 |
|------|----------|----------|
| 日志存储写入连续失败 ≥ 3 次 | 进入 STORAGE_FAULT，切换至内存缓冲，上报告警 | 存储恢复正常且后续写入成功 |
| 内存缓冲队列溢出（>5000条） | 丢弃最旧的日志条目，记录告警（优先保留执行结果与超时事件类日志） | 存储恢复后回写缓冲数据 |
| 日志存储空间低于 10% | 触发旧日志清理（保留最近 3 年），上报告警 | 空间释放至 20% 以上 |
| 日志存储空间低于 5% | 触发强制清理（保留最近 2 年），上报告警 | 空间释放至 15% 以上 |
| 查询请求超时（>500ms） | 返回部分结果并标记"查询未完成" | — |
| 紧急熔断 | 强制刷写内存缓冲至存储（若可用），暂停接收新事件 | 紧急解除后恢复 |
| 日志条目 HMAC 签名生成失败 | 使用空签名占位，标记"签名异常"，继续记录（不中断日志链路） | 签名密钥恢复 |

## 安全边界

| 规则编号 | 内容 |
|:---:|------|
| L-01 | 日志存储必须采用追加写模式（Append-Only），禁止任何模块修改或删除已落盘的日志条目 |
| L-02 | 每条日志条目必须包含全局单调递增序列号与 UTC 时间戳，确保时序不可篡改 |
| L-03 | 执行相关的关键日志（执行结果、超时事件、偏差事件）在存储空间不足时优先保留，不得被自动清理 |
| L-04 | 日志存储周期 ≥ 3 年，超期日志在空间不足时优先清理，但需在清理前发送告警通知 |
| L-05 | 日志 HMAC 签名用于内部完整性校验，发现签名不匹配的日志条目时应上报告警但不中断服务 |
| L-06 | 日志数据不得包含任何用户个人身份信息、认证凭证或敏感业务数据，仅记录工具调用的技术元数据 |
